# 🎒 초보 개발자를 위한 '북어 트립' 아키텍처 & 디버깅 학습서

안녕 주니어 개발자 친구들! 🍒 '북어 트립' 프로젝트에 온 걸 환영해. 
이 학습서는 이북(e-Book) 플랫폼을 빌딩하며 사용한 개념과 코드가 돌아가는 핵심 원리, 그리고 개발 과정에서 마주쳤던 아주 고약한 에러들과 그걸 해결해 나간 **진짜 디버깅 이야기**를 담고 있어. 

이론책에는 나오지 않는 생생한 실무 디버깅 스토리이니 코드를 열어두고 차근차근 읽어봐!

## 1. 뼈대 이해하기: 클라이언트-서버 모델

웹 서비스는 기본적으로 손님이 있는 테이블(클라이언트)과 요리를 만드는 주방(서버)으로 나뉘어 있어.

```
┌─────────────────────────────────┐                   ┌─────────────────────────────────┐
│     클라이언트 (손님 테이블)      │ ─── HTTP POST ──> │        서버 (요리 주방)         │
│  - 화면 그리기: index.html       │                   │  - 데이터 조립: api/index.py    │
│  - 스타일링: css/styles.css     │ <── JSON Response ─ │  - AI 로테이션 및 병렬 수집     │
│  - 동작 제어: js/app.js          │                   │                                 │
└─────────────────────────────────┘                   └─────────────────────────────────┘
```

1. **클라이언트 (Browser)**: 사용자가 보는 화면(`index.html`), 이쁘게 단장하는 옷(`styles.css`), 버튼을 눌렀을 때 작동하는 뇌(`app.js`)가 브라우저 안에서 돌아가.
2. **서버 (API Backend)**: 눈에 보이지 않는 먼 클라우드 컴퓨터에서 복잡한 생각과 정보 연산을 대신해 주는 두뇌 컴퓨터(`api/index.py`)야.
3. **HTTP 통신 (fetch)**: 브라우저가 "방콕 야시장으로 책 한 권 지어줘!" 하고 서버의 주소(`/api/generate`)로 편지(Request)를 보내면, 서버는 Gemini AI와 Unsplash 이미지 등을 취합해 하나의 완성된 이북 레시피 데이터(JSON Response)를 답장해 줘.

## 2. 화면이 꼬이지 않게 막는 장치: 이벤트 버블링과 전파 차단

HTML 요소들은 큰 상자 안에 중간 상자, 그 안에 작은 버튼이 들어있는 **부모-자식** 구조를 가져.

### ① 이벤트 버블링 (Event Bubbling) 현상
물속에 거품(Bubble)이 보글보글 위로 솟아오르듯, 자식 태그(예: 보관함 카드의 `삭제 X 버튼`)를 클릭하면, 클릭 이벤트가 자식에서 끝나지 않고 부모 태그(예: 보관함 `카드 몸통 전체`)로 연달아 솟구치며 둘 다 실행되는 현상이야.

### ② 우리가 겪은 문제와 해결책
버블링을 내버려 두면 사용자가 책 보관함 카드 우측의 '삭제 버튼'을 누르는 순간, 책이 지워짐과 동시에 부모 카드의 '책 열기' 함수가 작동하여 화면에 지워진 유령 책이 갑자기 열려버리는 UI 버그가 생겼어.
이를 막기 위해 삭제 이벤트 핸들러 안에 `evt.stopPropagation()` 한 줄을 장착했지!

```javascript
// [js/app.js] 삭제 단추 클릭 이벤트 핸들러
btnDelete.addEventListener("click", (evt) => {
  evt.stopPropagation(); // 👈 부모 카드로 거품처럼 올라가는 클릭 신호를 여기서 철저히 차단!
  
  // 책 삭제 로직 실행...
});
```
이 방어 코드가 부모 카드 몸통이 클릭되는 것을 물리적으로 막아주어, 삭제는 깔끔하게 삭제만 되고 창이 꼬이지 않게 되었어.

## 3. CSS 배경 슬라이싱을 이용한 양면 와이드 표지

북어 트립은 양면 펼침(Spread) 종이책 감성을 살리기 위해 책을 열자마자 첫 양면(1p & 2p)에 하나의 커다란 그림이 자연스럽게 이어져. 이를 위해 그림 파일을 조각내지 않고 CSS만으로 구현했어.

* **슬라이싱 원리 (`css/styles.css`)**:
  1. 두 개의 반쪽짜리 페이지(`cover-half-left`, `cover-half-right`)에 **완전히 동일한 커버 일러스트 이미지**를 배경으로 얹어.
  2. 배경 이미지 가로 너비를 실제 페이지 너비의 2배(200%)로 늘려놔 (`background-size: 200% 100%`).
  3. 그리고 왼쪽 페이지는 이미지의 왼쪽 반만 보여주고(`background-position: left center`), 오른쪽 페이지는 오른쪽 반만 보여주면(`background-position: right center`) 책을 펼쳤을 때 두 페이지의 그림이 자석처럼 정교하게 이어지게 돼!

## 4. 좌충우돌 디버깅 모험기: 북어 트립의 핵심 에러 해결 스토리 🚀

개발을 하며 맞닥뜨렸던 주옥같은 에러들과 이를 해결하며 배운 지혜들을 공유할게.

### 💥 스토리 A: Vercel의 10초 타임아웃 장벽 (Gateway Timeout 504)
* **어떤 에러였나?**
  처음 책을 집필하면 가끔 화면에 `504 Gateway Timeout` 에러가 뜨며 서버가 죽어버렸어.
* **왜 발생했나?**
  Vercel의 무료 Hobby 티어 서버리스 함수는 **10초** 내에 결과를 리턴하지 못하면 강제로 연결을 끊어버려. 
  그런데 4페이지 분량의 Unsplash 고화질 이미지를 인터넷에서 하나하나 차례대로 수집(`for` 루프 동기 수집)하고, AI 모델의 생각 시간이 겹쳐지면서 전체 동작 시간이 11~12초를 훌쩍 넘어가 버린 것이 원인이었어.
* **우리는 어떻게 해결했나? (`api/index.py`)**
  1. **병렬 수집 (`ThreadPoolExecutor`):** 파이썬의 멀티스레딩 도구를 활용해 4~6장의 Unsplash 이미지 검색 및 다운로드 요청을 4차선 도로처럼 **동시에(Parallel)** 수집하도록 고쳐 수집 대기 시간을 6초 ➡️ 1.2초로 대폭 단축시켰어.
  2. **5초 제한 로테이션:** 메인 AI(Gemini)에 장애가 생기거나 지연이 발생할 때 25초간 마냥 기다리던 것을 `5초` 타임아웃으로 강제 차단하고, Groq, Cloudflare AI 등 7대의 백업 AI 서버로 빠르게 Failover(미끄러지듯 교체)되도록 환경 변수 로테이션을 장착해 Vercel의 10초 한도를 지켰어.

### 💥 스토리 B: 서재와 챗봇의 먹통 소동 (CSS Specificity 인라인의 덫)
* **어떤 에러였나?**
  처음에는 잘 열리던 보관함 Drawer와 챗봇 창이 몇 번 껐다 켜고 나면 화면에서 미동조차 하지 않고 먹통이 되었어.
* **왜 발생했나?**
  처음 코드에서는 자바스크립트로 서랍장을 열고 닫을 때 `element.style.right = "0px"`, `element.style.right = "-350px"` 처럼 자바스크립트가 CSS 인라인 속성을 강제로 입력해 버렸어. 
  HTML의 CSS 우선순위 규칙에 따르면 **인라인 스타일(Inline Style)**은 스타일시트 파일(`.open`)에 정의된 클래스 규칙보다 훨씬 강력해서 덮어씌워지지 않아. 이 때문에 클래스 제어가 먹통이 된 거지.
* **우리는 어떻게 해결했나? (`js/app.js` & `css/styles.css`)**
  인라인 스타일 코드를 싹 다 제거하고, 오직 CSS 파일에 클래스(`.classList.add("open")`, `.classList.remove("open")`)를 추가 및 제거하는 정통 방식으로 통합했어. 클래스로 일원화하자 챗봇과 보관함 서랍장이 충돌 없이 부드럽게 작동하기 시작했어.

### 💥 스토리 C: 너무 긴 편지는 찢어져요 (공유 URL 파라미터 유실)
* **어떤 에러였나?**
  친구에게 내 책을 전달하기 위해 공유하기 링크를 보냈는데, 친구가 링크를 열면 책이 그대로 열리지 않고 또 로딩창이 뱅글뱅글 돌면서 아예 새로운 내용의 책이 생성되었어.
* **왜 발생했나?**
  도서 데이터 전체 JSON을 텍스트로 보존해서 공유하고 싶어서 링크 주소 뒤에 쿼리 파라미터(`?book_data=...`)로 길게 붙여 보냈어. 
  하지만 책 데이터는 크기가 매우 커서 Base64 인코딩을 하면 URL의 길이가 3~4KB에 육박했어. 일부 브라우저나 Vercel 프레임워크는 보안 및 네트워크 오버헤드를 막기 위해 URL이 너무 길면 자르거나 무시해 버려(`414 URI Too Long` 오류). 그 바람에 데이터가 유실되어 수신자는 책 정보를 읽지 못해 새로 생성하는 사태가 벌어졌던 거야.
* **우리는 어떻게 해결했나? (`js/app.js` & `api/index.py`)**
  데이터 전송 방식을 **URL Hash(#)** 방식으로 교체했어!
  - 예: `https://think-travel.vercel.app/api/share?title=...#<compressed_data>`
  - 주소창 뒤의 `#`(해시) 영역에 있는 글자는 **서버로 절대 날아가지 않고 브라우저에만 조용히 유지**돼. 
  - Vercel 서버는 긴 URL을 감지하지 않고 기본 페이지를 즉시 응답해주며, 브라우저가 로딩된 후 `window.location.hash`를 낚아채서 압축을 풀어내도록 구조를 설계했지. 서버 부하도 없고, 414 에러도 없이 0초 복원 공유가 가능해졌어.

### 💥 스토리 D: 지도 엽서가 너무 늦게 로딩돼요 (이미지 생성 지연)
* **어떤 에러였나?**
  책이 0초 만에 열리고 잘 동작하는데, 일정 페이지로 넘어가면 엽서 지도가 5~8초 동안 허옇게 비어있다가 늦게 뜨는 지연 현상이 눈에 띄었어.
* **왜 발생했나?**
  엽서 지도를 아름다운 빈티지 스타일의 AI 이미지로 직접 그려내도록 Pollinations AI에 동적 요청을 쐈는데, 디퓨전 모델이 실시간으로 그림을 그리려면 당연히 시간이 오래 소요되었기 때문이야.
* **우리는 어떻게 해결했나? (`api/index.py` & `js/app.js`)**
  1. **백엔드 선지연 자극 (Pre-triggering):** 브라우저에서 사용자가 책장을 넘겨서 지도를 보기 전, 즉 **상세 책 데이터 생성을 백엔드에서 수행하는 그 시점**에 `ThreadPoolExecutor` 스레드 중 하나가 Pollinations AI 맵 URL에 백그라운드로 먼저 콕 찔러서(`requests.get(map_url)`) 렌더링 작업을 지시해 뒀어.
  2. 사용자가 이북을 열고 구경하는 수초의 시간 동안 이미 지도는 백그라운드에서 완벽히 그려졌기 때문에, 지도 페이지에 도달했을 때는 캐싱된 이미지가 딜레이 전혀 없이 즉각 노출되게 되었어!
