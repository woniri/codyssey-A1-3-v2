# 🎒 초보 개발자를 위한 '북어 트립' 아키텍처 & 디버깅 완벽 정복 가이드

안녕 주니어 개발자 친구들! 🍒 
'북어 트립 (Book어 Trip)' 프로젝트에 온 걸 진심으로 환영해.

이 학습서는 단순히 코드를 훑어보는 가이드가 아니야. HTML, CSS, JavaScript, 그리고 Python FastAPI 백엔드가 서로 어떻게 맞물려 돌아가는지 **아키텍처의 밑바닥부터 완벽하게 파헤쳐 줄 교과서**란다.

더불어, 우리가 이 플랫폼을 만들면서 겪었던 아주 고약한 에러들과 그걸 해결해 나간 **진짜 디버깅 이야기**를 흥미진진하게 담았으니 코드를 열어두고 꼼꼼하게 읽어봐!

## 🗺️ 1. 북어 트립 전체 동작 원리 및 기술 뼈대

우리가 만든 서비스는 크게 두 구역으로 나뉘어 일해. 바로 사용자가 마주하는 **클라이언트(브라우저)**와 뒤편의 주방장인 **백엔드(FastAPI)**지. 이 둘이 어떤 기술로 이루어졌는지 알아보자.

```
┌────────────────────────────────────────────────────────┐
│               [프론트엔드 (클라이언트)]                 │
│  - index.html (구조)                                   │
│  - css/styles.css (종이 질감 필터, 3D 책 회전 모션)    │
│  - js/app.js (데이터 요청, LocalStorage 캐시, TTS 재생) │
└────────────────────────────────────────────────────────┘
                           │ 
                     HTTP 통신 (JSON)
                           │ 
┌────────────────────────────────────────────────────────┐
│                 [백엔드 (FastAPI)]                     │
│  - api/index.py (Gemini 2.5 Flash, Grounding 팩트체크)  │
│  - ThreadPoolExecutor (Unsplash 및 엽서 이미지 병렬화)  │
└────────────────────────────────────────────────────────┘
```

### ① HTML (3D 회전 책 뷰어 돔 구조)
책을 사실적으로 넘기기 위해, 우리는 `index.html` 안에 책(`class="book"`)과 여러 장의 페이지(`class="page"`)를 겹겹이 쌓아 두었어.
* **3D 입체 효과**: CSS의 `perspective: 1500px` 속성을 부모 상자에 주어, 눈앞에 진짜 책이 놓인 듯한 깊이감을 표현해.
* **책장 넘기기**: 자바스크립트로 페이지 뒤집기 버튼을 누르면, CSS의 `transform: rotateY(-180deg)` 속성이 발동되어 책장이 왼쪽으로 사르르 뒤집어지지!

### ② CSS (종이 질감 필터와 나무 책꽂이 디자인)
* **무광 종이 질감**: 최신 모니터의 쨍하고 차가운 느낌을 빼고 오래된 아날로그 종이 엽서 느낌을 주기 위해 CSS 필터 효과(`filter: sepia(12%) contrast(92%)`)를 오버레이하여 따스하게 눈을 정돈했어.
* **나무 책꽂이 (`bookshelf-wrapper`)**: 실제 나무 책장에 책이 꽂혀 있는 느낌을 연출하기 위해, 그라데이션 기법(`linear-gradient`)과 목재 텍스처 백그라운드를 활용해 브라우저 위에 나무의 질감을 얹었지.

### ③ JavaScript (API fetch 체인, LocalStorage 캐싱, Gzip deflate 압축)
* **fetch 통신**: 브라우저가 백엔드 주방에 비동기(Asynchronous)로 요리 요청을 보낼 때 쓰는 강력한 자바스크립트 엔진이야.
* **LocalStorage 캐시**: 생성한 책의 JSON 데이터를 보관함 서랍장(`localStorage.setItem('books', ...)`)에 영구 보존해 두었다가, 사용자가 다시 클릭하면 서버 통신 없이 0.1초 만에 책을 다시 복원해 내.
* **Gzip(Deflate) 압축**: 아주 긴 도서 레시피 데이터를 그대로 URL 주소 뒤에 실어 보내면 글자 수가 너무 많아 주소가 끊어지게 돼. 자바스크립트 내장 `CompressionStream`을 써서 데이터의 부피를 80% 이상 압축하는 똑똑한 설계를 담았어.

### ④ Python FastAPI 백엔드
* **Gemini 2.5 Flash**: 초고속 경량화 AI 모델로, 4페이지 분량의 수려한 도시 스토리 가이드북과 타임라인 일정을 빠르게 창작해 줘.
* **Web Search Grounding**: AI가 거짓 정보를 말하는 현상(Hallucination)을 막기 위해 Google 실시간 검색 엔진과 실시간 교차 검증을 강제했어.

## 🔄 2. 프론트엔드-백엔드 풀 라이프사이클 (Full Lifecycle) 흐름

사용자가 여행지 입력창에 **"오타루"**를 입력하고 검색 버튼을 누르는 순간부터 책이 눈앞에 활짝 펼쳐질 때까지의 모든 흐름을 단계별로 쫓아가 보자!

```
[1. 검색창 "오타루" 입력] ──> [2. 로컬스토리지 보관함 검사]
                                      │
                         (기존 제작 이력 없음)
                                      ▼
[4. 백엔드 /api/search 호출] <── [3. 화면에 모래시계 로딩창 가동]
         │
         ├──> [5. 대규모/단일 규모 자동 판정]
         │        - 오타루는 '단일 명소' ➡️ "SINGLE" 응답
         ▼
[6. 프론트엔드에서 즉시 /api/generate 요청]
         │
         ├──> [7. Gemini AI 상세 스토리 집필 (4페이지 제약)]
         ├──> [8. ThreadPoolExecutor 병렬 엔진 발동]
         │        - 1~4장 Unsplash 실제 풍경 이미지 다운로드
         │        - [핵심!] 오타루 감성 지도 엽서 이미지 사전 생성 요청 (Pollinations)
         ▼
[9. 완성된 책 데이터(JSON) 프론트엔드로 리턴]
         │
         ├──> [10. app.js 내 buildEbook() 실행]
         │        - HTML 돔을 조립하고 3D 책 프레임 가동
         │        - Leaflet.js 지형 맵핑
         ▼
[11. 책 열기 모션과 함께 "체리" 가이드 오디오 재생]
```

1. **검색:** 사용자가 "오타루"를 입력하고 서재 엮기 버튼을 클릭해.
2. **보관함 캐시 확인:** 자바스크립트는 브라우저 내부 보관함(`localStorage`)에 오타루 책 데이터가 있는지 먼저 조사해. 있으면 API 호출을 건너뛰고 10단계로 바로 워프(0.1초 로드)!
3. **로딩 애니메이션:** 캐시가 없으면, 화면 중앙에 모래시계 로딩창을 띄우고 사용자의 조작을 막아.
4. **1차 서재 판정 (`/api/search`):** 백엔드에 "오타루"가 큰 국가 단위(대규모 서재)인지 일반 도시(단일 도서)인지 판단해 달라고 요청해.
5. **서재 스킵:** 백엔드는 오타루가 단일 도시(`SINGLE`)임을 판정하고, 프론트엔드는 책꽂이 단계를 패스하고 곧장 6단계 책 생성으로 넘어가.
6. **2차 도서 생성 (`/api/generate`):** 백엔드로 본격적인 집필을 요청해.
7. **Gemini AI 스토리텔링:** Gemini AI가 정확히 4개의 챕터 본문과 오디오 낭독 텍스트, 그리고 경로 상세를 JSON 포맷으로 생성해 내.
8. **병렬 수집 스케줄러 (`ThreadPoolExecutor`):** 
   - 4개 장의 이미지 검색 키워드로 Unsplash에 이미지 주소를 동시에 호출해.
   - 동시에 사용자가 정의한 엽서 템플릿(오타루 고유의 웜 오렌지색, 운하길 특징 반영)으로 Pollinations AI에 지도 생성 명령을 병렬로 찔러 두어(Pre-trigger), 사용자가 탭을 전환하기 전에 이미지를 미리 완성해 둠.
9. **도서 전달:** 모든 텍스트와 풍경 이미지, 미리 렌더링된 지도 주소가 포함된 최종 레시피 데이터(JSON)가 프론트엔드로 복귀해.
10. **e-Book 빌드 (`buildEbook`):** `app.js`가 JSON 데이터를 바탕으로 HTML 구조를 생성해 3D 책장에 밀어 넣고, Leaflet.js를 사용해 실시간 약도 경로를 매핑해 탭에 장착해.
11. **도서 개방:** 서재에서 책이 스르륵 뽑혀 나와 화면에 입체적으로 열리며, Web Speech API 가이드 Cherry🍒의 낭독이 스피커로 흘러나와.

## 🎯 3. 이벤트 버블링과 돔(DOM) 제어 마스터

부모 태그 안에 자식 태그가 겹쳐 있을 때, 클릭 신호가 수면 위로 솟구치는 현상을 **이벤트 버블링**이라고 해.

```
┌──────────────────────────────────────────────┐
│  부모: 보관함 책 카드 (.cabinet-book-card)    │ 
│  (클릭 시: "해당 이북 책방을 화면에 열어라")     │ ▲ 클릭 전파 (Bubbling)
│  ┌────────────────────────────────────────┐  │ │
│  │ 자식: 삭제 단추 (btnDelete)            │  │ │ 
│  │ (클릭 시: "로컬스토리지에서 삭제해라")    │ ───┘
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

### 버블링 방어 실전 코드
만약 전파를 막지 않으면, 삭제 단추를 누르는 순간 부모 카드도 클릭된 것으로 인지되어 **"책을 로컬스토리지에서 지우는 동시에 지워져서 존재하지 않는 책방을 억지로 펼치려고 시도하는"** 기괴한 루프 버그가 꼬이게 돼.
그래서 아래와 같이 신호 전파를 차단해주어야 해:

```javascript
// 삭제 버튼에 클릭 이벤트를 줄 때
btnDelete.addEventListener("click", function(event) {
    // 1. 거품 전파를 물리적으로 쾅! 가로막아버리기
    event.stopPropagation(); 
    
    // 2. 보관함에서 삭제하는 로직만 안전하게 수행
    deleteBookFromStorage(bookId);
});
```

## 🎨 4. CSS Specificity (우선순위)와 인라인의 덫

CSS 속성이 겹칠 때 브라우저는 어떤 속성을 화면에 보여줄지 점수를 계산해. 이것을 **우선순위(Specificity)**라고 부르지.

* **인라인 스타일 (Inline Style):** HTML 태그 안에 직접 적는 `style="..."`은 1000점짜리 왕이야.
* **아이디 셀렉터 (`#id`):** 100점.
* **클래스 셀렉터 (`.class`):** 10점.

### 우리가 마주친 재앙과 리팩토링
자바스크립트 파일(`app.js`)에서 챗봇 창이나 서랍장 캐비닛을 움직일 때 인라인 스타일인 `.style.right = "0px"`을 무심코 대입했다가 대참사가 발생했어.
나중에 스타일시트(`.css`) 파일에 아무리 `.cabinet-drawer.open { right: 0; }` 클래스 규칙(10점)을 추가해도, 이미 자바스크립트가 인라인 스타일(1000점)로 고정 지시를 내렸기 때문에 클래스가 교체되어도 화면이 전혀 꼼짝도 하지 않았던 거야!

* **디버깅 방향:** 
  모든 인라인 스타일 코드를 제거하고, CSS 파일에 서랍장의 기본 닫힘 속성과 `.open` 클래스가 붙었을 때의 열림 속성을 미리 다 적어둔 뒤, 자바스크립트는 오직 `.classList.add("open")`과 `.classList.remove("open")` 클래스 꼬리표만 떼었다 붙였다 하도록 역할을 분담하여 우선순위 엉킴을 완벽히 해결했어.

## 🚀 5. 좌충우돌 디버깅 모험기: 북어 트립의 6대 핵심 에러 해결 스토리

개발자들이 밤을 지새우며 해결한 주옥같은 6가지 디버깅 연대기야.

### 💥 스토리 A: Vercel의 10초 타임아웃 장벽 (Gateway Timeout 504)
* **현상:** 책 만들기를 누르면 가끔 화면이 멈추고 서버가 먹통이 됨.
* **원인:** Vercel 무료 서버는 연산 대기가 **10초**를 넘으면 504 Gateway 에러를 내며 강제 중단해. AI의 생각 시간과 4장 이상의 고화질 관광지 사진을 하나씩 검색해 오는 시간(동기 수집)이 누적되어 11초가 넘게 걸린 것이 문제였어.
* **해결:** `ThreadPoolExecutor`를 탑재하여 여러 장의 사진 검색과 엽서 지도 호출을 멀티스레드로 **동시에 병렬(Parallel) 수집**하여 연산 대기 시간을 6초 ➡️ 1.2초로 대폭 압축하고 AI 타임아웃 백업망을 로테이션시켜 장벽을 극복했어.

### 💥 스토리 B: 서재와 챗봇의 먹통 소동 (인라인의 덫)
* **현상:** 보관함 창과 챗봇 창이 클릭을 반복하면 굳어버림.
* **원인:** JS에서 인라인 스타일로 좌표를 지정한 탓에 CSS 스타일시트 클래스 지시보다 점수가 높아 덮어쓰기 오동작을 일으킴.
* **해결:** JS는 인라인 스타일 지정을 싹 다 중단하고, 오직 CSS 클래스(`.classList.toggle("open")`)의 제어만 전담하여 우선순위를 정리함.

### 💥 스토리 C: 유령 책방 소동 (이벤트 버블링)
* **현상:** 책 삭제 X 단추를 누르면 책이 지워진 뒤 화면에 빈 유령 이북이 억지로 펼쳐짐.
* **원인:** 삭제 버튼의 클릭 신호가 부모 카드 전체 영역으로 타고 올라가 이북 열기 함수가 연달아 실행됨.
* **해결:** 삭제 버튼 핸들러에 `evt.stopPropagation()`을 선언하여 거품 전파를 물리적으로 튕겨내 해결함.

### 💥 스토리 D: 너무 긴 주소는 찢어집니다 (URL 길이 제한 414 에러)
* **현상:** 공유 링크를 카카오톡으로 받아 열었는데 책이 복원되지 않고 로딩창이 돌며 책을 새로 쓰기 시작함.
* **원인:** 전체 도서 데이터를 Base64 글자로 바꿔 URL 뒤 쿼리 파라미터(`?book_data=...`)로 길게 보냈더니 주소 폭이 3KB가 넘어 서버와 브라우저 단에서 뒷부분이 뎅강 잘려 유실된 것임.
* **해결:** 서버로 전송되지 않고 오직 클라이언트 브라우저 단에서만 보존되는 **URL 해시(#)** 기법을 도입하여 `#<압축데이터>` 방식으로 전송해 서버 414 에러 및 데이터 유실을 100% 원천 차단함.

### 💥 스토리 E: 지도가 늦게 인쇄되어 검은 구멍이 생겨요 (이미지 렌더링 지연)
* **현상:** 책장을 넘겨 일정 페이지에 도착했을 때, 엽서 지도가 아직 안 그려져서 5~8초간 빈 화면이 뜸.
* **원인:** AI 디퓨전 모델이 실시간으로 동양풍 지도를 그리려면 어쩔 수 없이 수초의 시간이 소요됨.
* **해결:** 이북이 완성될 즈음 백엔드 스레드가 지도의 주소(`mapImageUrl`)를 미리 GET 통신으로 한 번 콕 찔러두는 **사전 자극(Pre-triggering) 기법**을 장착했어. 사용자가 책을 넘겨 지도 탭을 볼 때쯤이면 이미지가 완성되어 캐싱 상태이므로 지연 없이 즉시 인쇄되어 나옴.

### 💥 스토리 F: 삿포로 책을 만들었는데 왜 방콕 야시장이 나오죠? (템플릿 하드코딩의 늪)
* **현상:** 오타루나 삿포로 가이드북을 엮어 펼쳐도 엽서 지도 영역에 계속 방콕 툭툭과 태국 야시장 천막 일러스트가 잘못 섞여서 생성됨.
* **원인:** 이전 지도 프롬프트 생성기(`generate_map_postcard_prompt`) 안에 방콕 야시장의 디자인 규칙과 태국 관련 고정 텍스트("망고 향과 불빛", "Tuk Tuk")가 고스란히 하드코딩으로 고정되어 다른 여행지 생성 시에도 방콕의 피사체들이 섞여 나온 것임.
* **해결:** 사용자가 설계한 고유 템플릿과 세부 데이터(삿포로, 오타루, 방콕 분기 매퍼)를 백엔드에 이식하여, 대상 여행지 영문명에 맞게 고유 감성 컬러와 특징(오타루의 운하길과 항구 실루엣 등)을 분기 적용하도록 동적 매핑했어. 
* 또한, 동그란 핀포인트(`location pin`)와 핀들을 잇는 굵은 주행 점선 경로(`Bold dashed walking route line`)를 프롬프트에 명문화하여 AI가 인쇄된 손그림 지도 형태로 빨간 핀들과 다채로운 점선을 그리도록 고쳐서 각 책에 완벽히 매치되는 고유 지도 카드를 완성했어!
